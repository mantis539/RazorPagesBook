## Контекст
Для забезпечення чистоти коду та можливості тестування ми переходимо від монолітної структури всередині PageModels до розподілу відповідальності за шарами.

## Рішення
Ми впроваджуємо трирівневу структуру:

1. **API/Pages Layer:** Приймає запити. Використовує DTO для вводу/виводу. Не містить бізнес-логіки.
2. **Service Layer:** Реалізує бізнес-сценарії (Use-cases). Тільки тут дозволено мапінг між DTO та Domain моделями.
3. **Domain Layer:** Містить чисті сутності бази даних та базові правила валідації об'єктів.

## Напрямок залежностей
Залежності мають бути спрямовані тільки **зверху вниз**:
- API знає про Service.
- Service знає про Domain.
- **Заборонено:** Domain не може звертатися до Service. API не може звертатися до БД напряму (минаючи Service).

## Наслідки
- **Плюси:** Легко замінити базу даних або інтерфейс незалежно один від одного.
- **Мінуси:** Потрібно писати більше коду (створювати DTO та сервіси навіть для простих операцій).
